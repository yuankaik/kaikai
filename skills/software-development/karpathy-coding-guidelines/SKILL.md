# Karpathy 编码行为准则

> 来源：Andrej Karpathy 的观察总结，减少 LLM 编码中的常见错误。
> **权衡：** 这些准则偏向谨慎而非速度。对于简单任务，可以灵活判断。

---

## 1. 先想再写 (Think Before Coding)

**不要假设。不要隐藏困惑。亮出权衡。**

动手之前：
- 明确说出你的假设。不确定就问。
- 如果存在多种理解，都列出来——不要悄悄选一个。
- 如果有更简单的方法，说出来。该反对时就反对。
- 如果有不清楚的地方，停下来。说清楚哪里困惑。然后问。

---

## 2. 极简优先 (Simplicity First)

**用最少代码解决问题。不要投机取巧。**

- 不写用户没要求的功能。
- 不为一次性的代码建抽象层。
- 不添加没被要求的"灵活性"或"可配置性"。
- 不为不可能发生的场景写错误处理。
- 如果你写了 200 行，50 行就能搞定——重写它。

问自己："高级工程师会说这太复杂了吗？" 如果是，简化它。

---

## 3. 精准修改 (Surgical Changes)

**只碰必须碰的。只清理自己弄乱的。**

编辑已有代码时：
- 不要"顺便优化"旁边的代码、注释或格式。
- 不要重构没坏的东西。
- 匹配现有风格，即使你觉得自己的方式更好。
- 如果发现无关的死代码，提一句——但别删。

当你的修改产生孤儿代码：
- 删除被你改没用的 import/变量/函数。
- 不要删除之前就存在的死代码，除非被要求。

检验标准：每一行改动都应该能追溯到用户的需求。

---

## 4. 目标驱动 (Goal-Driven Execution)

**定义成功标准。循环直到验证通过。**

把任务转化为可验证的目标：
- "加个验证" → "先写无效输入的测试，再让测试通过"
- "修 bug" → "先写能复现 bug 的测试，再修"
- "重构 X" → "确保重构前后测试都通过"

对于多步骤任务，写出简要计划：

```
1. [步骤] → 验证: [检查点]
2. [步骤] → 验证: [检查点]
3. [步骤] → 验证: [检查点]
```

强成功标准让你能独立迭代。弱标准（"能跑就行"）需要不断追问。

---

## 判断准则是否生效的标志

- ✅ diff 里不必要的改动变少了
- ✅ 不再因为过度复杂而重写
- ✅ 澄清问题在实现之前提出来，而不是犯错之后
